# 9. OpenTelemetry + Jaeger for Distributed Tracing and Log Correlation


Date: 2026-03-05

## Status

Accepted

## Context

Debugging a financial microservice requires observability beyond simple application logs. When a withdrawal fails or behaves unexpectedly, an operator needs to:

1. **Correlate log lines** from disparate structured log records into a single request timeline.
2. **Span individual operations** (MySQL query duration, Redis lookup, SMTP call) to identify latency bottlenecks.
3. **Propagate context** across future service-to-service calls (if the architecture expands).

Options considered:

| Approach | Notes |
|---|---|
| Application logs only | Good for errors; no timing information; hard to correlate across middleware layers |
| Zipkin | CNCF project; lighter UI; smaller ecosystem than Jaeger |
| **Jaeger** | CNCF project; battle-tested; rich UI; supports OTLP ingestion; Docker image available |
| Datadog / New Relic APM | SaaS; excellent tooling; requires internet access; cost per host |
| OpenZipkin via OpenTelemetry | Can use OpenTelemetry SDK targeting Zipkin exporter |

For the SDK layer:

| Approach | Notes |
|---|---|
| Vendor-specific agents (Datadog agent, Jaeger client) | Lock-in; separate SDK per vendor |
| **OpenTelemetry SDK** | Vendor-neutral; CNCF standard; single SDK for traces, metrics, logs; plug exporter for any backend |

## Decision

Use the **OpenTelemetry PHP SDK** (`open-telemetry/opentelemetry-php`) with the **OTLP/HTTP exporter** pointing to a local **Jaeger** instance.

Architecture:
- `OTelTracer` adapter (`app/Infrastructure/Observability/`) implements the `TracerInterface` domain port and wraps the OpenTelemetry `TracerProvider`.
- The **OTLP/HTTP exporter** sends spans to Jaeger's collector on `http://jaeger:4318/v1/traces` (configurable via `OTEL_EXPORTER_OTLP_ENDPOINT`).
- `TraceContextProcessor` (Monolog processor at `app/Infrastructure/Log/`) enriches every log record with the active `trace_id` and `span_id`, enabling log→trace correlation in Jaeger.
- The `X-Request-Id` header (generated by `RequestIdMiddleware`) is also injected into log records and stored as a span attribute, allowing correlation between HTTP access logs and trace data.

Jaeger is run as an all-in-one Docker container (port 16686 for the UI, port 4318 for OTLP/HTTP ingestion).

## Consequences

**Positive:**
- OpenTelemetry is the CNCF standard; switching from Jaeger to Zipkin, Datadog, or Grafana Tempo requires only changing the exporter configuration.
- All HTTP requests, database queries (via Hyperf DB listener), and Redis operations produce spans visible in Jaeger's flame graph UI.
- Log-trace correlation allows operators to jump from a log line directly to the full trace in Jaeger.
- `TracerInterface` port means the domain and use cases do not depend on OpenTelemetry; tracing is an infrastructure concern.

**Negative / Risks:**
- The OTLP exporter is currently synchronous; exporting spans adds a small latency to each request. An async/batch exporter (`BatchSpanProcessor`) would eliminate this at the cost of potential span loss on crash.
- Jaeger's all-in-one Docker image stores traces in memory only; traces are lost on container restart. For persistent storage, Jaeger should be backed by Elasticsearch or Cassandra.
- The current setup does not export metrics (only traces). Extending to metrics (request counts, latency histograms) would require adding an OpenTelemetry metrics pipeline.
